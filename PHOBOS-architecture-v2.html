01 · Three-Brain Architecture
PHOBOS CORE
Backend Process · Node.js server · DuckDB · tree-sitter
HTTP / SSE server Task queue poll loop Dispatch controller File tool executor Build subprocess Skill runner Attempt counter All DuckDB I/O
SAYON
Coordinator · Highest backendScore device (CUDA=3, Metal=2, discrete Vulkan=1) · port 52625
Compute-primary assignment 256k ctx 28–50 TPS Qwen3-8B / Llama3.x Intent Classification Prompt Reformation Full Tool + File Capability Live Reviewer Intermediate Planner
SEREN
Reasoning Engine · Highest VRAM device (GPU or unified memory) · port 11434
VRAM-primary assignment 128k ctx Extended thinking Qwen3-30B / DeepSeek-R1 Technical Authority Primary Planner Deep Execution Final Validation Final Delivery
Dynamic peer model — TWIN SUN. SAYON and SEREN are colleagues with complementary strengths — not a hierarchy. SEREN is the technical authority: it plans, reasons deeply, decides the work split, and does final validation. SAYON is practical and fast: it interprets the user, manages context, and takes on whatever SEREN routes to it, including file writes, tool calls, and execution tasks. Both have full capability. CORE is the only brain that never generates text — it dispatches, executes infrastructure, and persists state exclusively.

Hardware assignment (two-sort): SEREN is assigned the highest-VRAM device (favours deep reasoning quality). SAYON is assigned the highest-backendScore remaining device (favours compute throughput: CUDA=3, Metal=2, discrete Vulkan=1, UMA/index≥100=0). On single-GPU machines, SAYON and SEREN share the same device.

04 Execution Loop SAYON orchestrates · SEREN executes
SAYON — dispatch & monitor
Polls task_queue in DuckDB. Dispatches next eligible task to correct executor
Accepts new user queries mid-execution → plans them → appends to task_queue tail as SAYON or SEREN tasks
If idle: reads SEREN's live thinking stream → emits progress updates and optional suggestions to user
Executes its own tasks: reformulation, context prep, tool calls, quick lookups
Reviews SEREN output: APPROVE / NEEDS_REVISION / REJECT
On APPROVE: syntax_validate → apply file tools → emit file_panel → mark task complete
On NEEDS_REVISION (<3): inject guidance + error logs → re-dispatch to SEREN
SEREN finished segment → SAYON can re-queue for further SEREN passes
SEREN — deep execution
Receives distilled task prompt from SAYON (no raw docs)
Extended thinking → think tokens stream to Thinking Column in real time
Issues file tool calls: write_file, replace_lines, insert_lines, delete_lines, read_file
Read→Act cycle: read_file for line numbers → surgical edits. Max 3 read cycles/attempt
Output tokens → Thinking Column (pending review)
Max 3 attempts per task before FAIL, reasons recorded
Shared State (DuckDB)
task_queue: ordered tasks, executor tag, status, dependencies
dispatch_log: tokens, latency, attempt #, review score per task
workspace_files: invalidated + re-indexed after each write
message_events: persisted SSE events for history replay
SAYON and SEREN share read/write access via DB — no direct IPC

03 · Task Queue Model
■ task_queue (DuckDB)
1 Read auth.ts + jwt.ts, extract interface shapes SAYON
2 Rewrite middleware.ts: replace session check with JWT verify SEREN
3 Create src/auth/jwt.ts utility (sign + verify) SEREN

Queue Schema

thread_idfk → threads
seqinteger, ordered
executor'sayon' | 'seren'
statuspending | running | done | failed

file_targetsstring[] — scoped file list
depends_onseq[] — dependency chain
planned_by'seren' | 'sayon'
resultjson — output + review score
CORE drives the poll loop. SAYON never polls the queue — CORE does. SAYON's role is to plan tasks and write them to DuckDB. CORE reads the queue, resolves dependency chains, and dispatches to the correct model endpoint. When SEREN's queue is active and a new user query arrives, SAYON plans additional tasks and writes them to the queue tail. CORE detects the new entries and dispatches eligible ones immediately — SAYON tasks that are dependency-free can run on the NPU in parallel with whatever SEREN is executing.

05 · Recursive Pagination — Default for All Reads & Writes
Recursive Read — Arbitrarily Large Documents
1
SAYON calls token_count(document). If ≤ SEREN window → single dispatch, done. If > → begin pagination. SAYON
2
paginate(doc, window×0.75, overlap=0.15) at AST/MD logical boundaries. 15% overlap between adjacent chunks. SAYON prepends extraction instructions + position/total to every chunk — SEREN is amnesiac between chunks. SAYON
3
CORE dispatches each chunk to SEREN sequentially. SEREN returns structured summary per chunk. SAYON appends to level_summaries[]. SEREN
4
token_count(level_summaries). Fits in one context? → FINAL SUMMARY, done. Still too large? → treat level_summaries as the new document and recurse from step 2. SAYON recurse
5
Each recursion level uses more abstract instructions: "summarise the summaries, preserve key decisions and constraints." Recursion terminates when result fits.
6
FINAL SUMMARY is what gets injected into the task prompt — this is "what the AI read" from the original document, regardless of how many recursion levels were needed. → ingest_context
Paginated Write — Large File Generation (flat, not recursive)
1
SAYON uses repo_map to divide target file into logical write sections: by class, exported function group, or module. Each section must fit in one SEREN context. SAYON
2
Dispatch section[0]: "Write this section. Interface context: [relevant types from adjacent sections]" SEREN
3
SAYON reviews section[0]. syntax_validate. APPROVE → CORE applies. FAIL → retry section only (not full file). Max 3 per section. SAYON
4
Dispatch section[1]: includes confirmed section[0] as interface reference. Sequential — each section confirms before next dispatches.
5
chunk_assemble(confirmed_sections[]) → final file. workspace_index_update triggers repo_map regen. SAYON
Read → Act Cycle (Surgical Edits to Existing Files)
1
SEREN issues <read_file path="…"/>. CORE detects all-read turn → executes read → returns full numbered content to SEREN as continuation. SEREN
2
SEREN issues surgical write: <replace_lines start_line="42" end_line="48">. No guessing — exact line numbers from the read.
3
CORE enforces max 3 read cycles per attempt. This is separate from paginated read — it is for precision execution, not comprehension. CORE guard
SAYON Mid-Run Planning (New User Query During Execution)
1
User sends new message while CORE's execution loop is running. CORE routes HTTP request → SAYON immediately. CORE routes
2
SAYON reads current task_queue state + ingest_context from DuckDB. Plans new tasks. Simple → executor=SAYON. Complex → executor=SEREN, depends_on=[last SEREN seq]. SAYON
3
SAYON writes new tasks to queue tail via task_queue_write. CORE detects new entries on next poll cycle. DuckDB
4
Unblocked SAYON tasks → CORE dispatches immediately to NPU (parallel with SEREN). SEREN tasks queue behind current SEREN work. User sees coordinator bubble. CORE dispatches
Paginated Read — Oversized Document
1
SAYON receives file, calls token_count(content) SAYON
2
Exceeds limit → paginate(content, maxTokens, 0.12) → chunks[] split at AST/MD boundaries SAYON
3
SAYON prepends task instructions to chunk[0] → dispatches to SEREN SEREN
4
SEREN returns structured findings for chunk[0]. SAYON appends to Complete Context DuckDB
5
Repeat for chunk[1..N]. Each chunk includes 10–15% overlap of prior chunk's end. Instructions prepended every time (SEREN is amnesiac between chunks)
6
SAYON assembles all findings → token_count final package → condense if needed → task dispatch SAYON
Paginated Write — Large File Creation
1
SAYON divides target file into logical sections: by function, class, or block boundary SAYON
2
Dispatch section[0] task to SEREN with precise section spec + surrounding interface context SEREN
3
SEREN outputs section[0] using write_file / append_file. SAYON reviews + applies SAYON
4
syntax_validate on section output. Pass → next section. Fail → retry with error SAYON
5
Repeat for section[1..N]. Each dispatch includes assembled prior sections as interface reference
6
chunk_assemble(sections[]) → validated final file written to workspace. workspace_index_update triggered SAYON

1
SEREN issues <read_file path="src/auth.ts"/> — backend returns full numbered content SEREN
2
Backend detects all-read turn → executes read → feeds numbered lines back as continuation Backend
3
SEREN issues precise write: <replace_lines path start_line="42" end_line="48"> SEREN

Max 3 read cycles per attempt. Prevents infinite read loops SAYON guard
SAYON Mid-Run Planning (New User Query)
1
User sends new message while SEREN task queue is executing SAYON
2
SAYON reads new input + current task_queue_read to understand in-progress context SAYON
3
SAYON plans new tasks: simple ones it handles itself, complex ones tagged executor=seren with depends_on last active SEREN task

Can use any system tool, script, or binary available on the machine. Full process isolation — runs as subprocess in workspace dir.

Skill Invocation Flow
Discovery (session start)
SAYON calls skill_list() → reads all installed skill manifests + trigger hints → builds available-tool summary → injected into SAYON's system context. Skills become part of SAYON's planning vocabulary automatically.
Auto-trigger (SAYON planning phase)
During task decomposition, SAYON checks user intent against skill trigger hints. Match → task tagged with skill_id. Skill task can be executor=sayon (fast, NPU-side parameter prep + subprocess call) or executor=seren (SEREN decides params with reasoning).
Execution
skill_invoke(skill_id, params) → skill runner subprocess → JSON result → injected as structured context into next task or returned directly to user. Build failures, lint errors, test results all flow through the same review loop as file edits.
Installation
User drops a .phobos-skill package in a local skills directory or provides a URL. SAYON detects it (or user explicitly installs via copilot / command). skill_install() validates manifest, registers to DuckDB skills table, immediately available to both models. No restart needed.

09 · Image Generation Pipeline
Generation Lifecycle — 9 Phases (owned by generateWithFlux())
1
Preliminary buildSdConfig() — device identification only. No VRAM queries yet. Establish which GPU handles image gen.
2
snapshotServerOnDevice() — find which LLM server (SAYON or SEREN) is currently running on the target device.
3
stopServer(snap.role) — SIGTERM the identified server, wait for process exit.
4
VRAM settle poll — query nvidia-smi every 500ms until free VRAM > 5 GB or 5s timeout (~1700ms typical).
5
Final buildSdConfig() — real config with post-stop VRAM. T5 encoder selected: ≥16 GB → Q8_0 (5.06 GB), ≥10 GB → Q4_K_M (2.9 GB), <10 GB → Q3_K_M (2.3 GB).
6
Spawn sd-cli — subprocess with assembled config. SSE stream opens to frontend immediately.
7
Monitor output — parse sd-cli stdout for sampling progress. Emit image_status SSE events per step.
8
Completion — write PNG to workspace images/ directory. Emit image_complete SSE with path. PNG never injected into LLM text context.
9
Restart server — restart the LLM server that was stopped in Phase 3. SEREN resumes on its device.
VRAM / VMM Rules
·
Use total VRAM for model selection — not free VRAM. VMM-capable NVIDIA GPUs can exceed the free VRAM reported by nvidia-smi. Model selection must be based on card capacity.
·
Use post-stop free VRAM for T5 selection — T5 encoder tier is chosen after the LLM server has been stopped and VRAM has settled. This is the only per-generation VRAM query that gates config.
·
backendScore for GPU selection — CUDA discrete = 3, Metal = 2, discrete Vulkan = 1, UMA or index ≥ 100 = 0. Prevents AMD iGPU (large UMA pool) from outbidding a discrete CUDA card.
·
Rule: caller never pre-builds config. generateWithFlux() owns the full lifecycle. No caller may snapshot state or build config before calling — doing so will use stale pre-stop VRAM figures.
·
Workspace routing: images/ is in IGNORE_PATTERNS — PNG files never appear in LLM text context. Images load via GET /api/threads/:id/workspace-media on thread open.

Classifier Design Rule

The prompt that powers IntentClassifier must be domain-agnostic. SAYON infers structure from user intent and file types — not from a built-in assumption that "this is a coding assistant".

12 · Autarch · Halcyon Distributed Training System
Autarch — Company and Platform Identity
·
PHOBOS is an Autarch product. Autarch's focus: decentralized, community-driven model development. PHOBOS users are the primary source of high-quality training data for Autarch models.
·
The reasoning quality available through PHOBOS's dual-engine architecture makes it uniquely suited to generate elite training data — better than single-model pipelines can produce.
·
First model under development: Halcyon. Reasoning-focused. Trained on validated PHOBOS reasoning generations.
Contribution Flow — User Opt-In
1
User selects a model from the Contribute to dropdown (e.g. Halcyon). This engages reasoning training mode for that session or turn.
2
SAYON and SEREN process the query as normal — TWIN SUN reasoning generates richer, validated chains than single-model approaches.
3
PHOBOS CORE captures the reasoning generation (thinking tokens + output tokens + SAYON review + SEREN validation). Validates quality locally before buffering.
4
Validated generations written to a local buffer on the user's machine. Buffer is flushed to a central staging buffer on Autarch's infrastructure at configurable intervals (never blocking the user's workflow).
Central Aggregation Pipeline
1
Central staging buffer receives contributions from all PHOBOS users opted into a given model.
2
Aggregation server deduplicates, normalises, and runs automated quality checks across the combined dataset.
3
Human or automated review flags edge cases. Cleaned dataset feeds into the training pipeline for the target model (Halcyon).
4
Model versioning: each training run is tagged to the dataset snapshot. Users who contributed to a run are credited in release notes.
Local Validation — Quality Gate Before Contribution
·
PHOBOS CORE validates each generation before buffering. Checks: reasoning coherence (does thinking lead to output?), task completion, SAYON review result (APPROVE only — HARD_FAIL outputs are never contributed).
·
Minimum quality threshold is configurable per model target. Halcyon starts with a strict threshold given the reasoning focus.
·
User can review their local buffer before flush. Opt to exclude individual turns. Full data ownership.
·
Implementation is elaborated post-core-development. Architecture hooks for contribution capture are built into PHOBOS CORE from the start — no retrofit needed.
